# Kion environment copy

Copy a Kion install's **billing sources, OUs, funding sources, projects, budgets,
and scopes** to another install. The two installs have different numeric ids, so
the tool stores cross-references (permission schemes, owners, accounts) **by name /
stable key** and rebuilds the id links on import.

## How it works

```
export  ─►  snapshot.json  ─►  import (plan)  ─►  import --apply  ─►  target
            name-based refs     diff vs target     create the diff      + id-map.json
                                (read-only)                              (state file)
```

`import` is a **terraform-style reconcile**, not a blind create. It diffs the
snapshot against the target's current state and only acts on the difference, so
`--apply` is safe to run repeatedly. Each entity gets one of:

| Action | Meaning |
|--------|---------|
| **create** | not in state and not on the target → POST it |
| **adopt** | not in state but an entity with the same name/scope already exists → record the mapping (like `terraform import`); **no write** |
| **in-sync** | already in the state file and present on the target → nothing to do |
| **recreate** | in state but the target id is gone → POST again |
| **drift** | present but some fields differ → **reported only**, never modified |

- **`id-map.json` is the state file** — it maps each snapshot (source) id to the
  target id. Re-running reads it to know what is already managed.
- **Adoption** keys on natural identity (name within parent/OU scope), so pointing
  at a target that already contains the environment yields **zero creates instead
  of duplicates** — even with no state file.
- **Dependency order**: billing sources → OUs (parents first) → funding sources →
  projects → budgets → scopes.
- **Permission schemes** match by name, else the stock per-type default scheme,
  else `DEFAULT_PERMISSION_SCHEME_ID`.
- **Owners** match by email / group name; missing ones are dropped.
- The source **root OU** maps onto the target's existing root — no duplicate root.

## Setup

```sh
cd kion-env-copy
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

Get an API key in Kion: **User Profile → App API Keys → create**.

## Copy an environment

```sh
# 1. Configure source and target
cp .env.example .env.source   # fill in SOURCE KION_URL + KION_API_KEY
cp .env.example .env.target   # fill in TARGET KION_URL + KION_API_KEY + DEFAULT_PERMISSION_SCHEME_ID

# 2. Export from source
python kion_copy.py export --env-file .env.source            # -> snapshot.json

# 3. Plan the reconcile (read-only — shows create/adopt/in-sync/drift)
python kion_copy.py import --env-file .env.target

# 4. Apply it (creates only the diff; writes id-map.json state)
python kion_copy.py import --env-file .env.target --apply

# 5. Re-run any time — already-reconciled entities show "in sync", nothing changes
python kion_copy.py import --env-file .env.target --apply
```

Always review the plan first — it lists what would be created, what would be
adopted, any drift on existing entities, and every reference that falls back to a
default or gets dropped.

## Files

| File | Purpose |
|------|---------|
| `kion_copy.py` | CLI (`export` / `import`) |
| `kion/client.py` | HTTP client (auth, retry, response unwrapping) |
| `kion/config.py` | `.env` loader |
| `kion/export.py` | source install → `snapshot.json` |
| `kion/import_.py` | reconcile: plan/apply, adoption, id remapping, drift |
| `snapshot.json` | exported environment (generated) |
| `id-map.json` | **state file**: source→target id map (written by `--apply`) |

## Notes / limitations

These follow from what the Kion **read** API actually exposes (verified against a
live install — several fields that `POST` requires are not returned by `GET`):

- **OUs** copy faithfully: hierarchy, description, permission scheme (matched by
  name), and owners (users by email, groups by name; missing ones skipped).
- **Funding sources**: the read API does **not** return a funding source's OU or
  permission scheme. On import each funding source is therefore placed on the
  **target root OU** and assigned the **Default Funding Source Permissions
  Scheme** (or `DEFAULT_PERMISSION_SCHEME_ID`). Name, amount, and dates copy
  exactly. Re-parent them in the UI afterward if needed.
- **Projects**: the read API does not return a project's permission scheme or
  owners, so projects get the **Default Project Permissions Scheme**. Name,
  description, OU, `auto_pay`, and default region copy exactly.
- **Budgets** copy with their monthly rows (datecode + amount) and funding-source
  links remapped to the new funding source ids. Rows without a funding source are
  kept as plain spend-plan amounts. Budget names are auto-generated by Kion from
  the timeframe, so a budget's identity is `(scope, start_datecode, end_datecode)`,
  not its name. A budget whose timeframe isn't fully covered by its rows, or whose
  funding source lacks sufficient funds, is rejected by Kion and reported as failed
  — with the specific cause named: the missing month(s) for a coverage gap, or the
  over-subscribed funding source (`$X allocated across N budgets vs $Y available`)
  for insufficient funds. These reflect inconsistent/over-subscribed **source**
  data, not import bugs.
- **Billing sources** copy as **non-functional shells**. The read API exposes
  connection config (bucket/region/prefix, org id, account number, etc.) but never
  secrets or trust (`key_secret`, AWS `linked_role`, OCI `private_key`, Azure/GCP
  credentials), so they're recreated with `skip_validation` and placeholder values
  the customer replaces on the target. Only **custom, AWS, and OCI** have an API
  create path; **GCP, Azure, and Anthropic** are exported but skipped (they require
  a prerequisite service account or a provider registration flow). Adopted by name.
- **Scopes** (`/beta/scope`, project cost-allocation rules) copy with their
  name/dates and resource/tag/logic criteria. Their **account criteria are remapped
  by `account_number`**; accounts missing on the target are dropped. Because Kion
  requires every scope to reference **at least one existing account**, a scope is
  **skipped** if none of its accounts exist on the target — so set up cloud accounts
  on the target before expecting scopes to import.

### Owners and permission schemes

- Owners are matched by email / group name; unmatched ones are dropped. Kion
  requires **every OU/funding/project to have at least one owner**, so when none is
  captured or resolved, the **user running the import** is assigned as owner
  (reported in the summary).

### Install differences handled automatically

- **API path**: hosted installs serve the API under `/api`; an app hit directly
  (e.g. `http://localhost:8081`) serves it at the root. Set `KION_API_PREFIX=`
  (empty) for the latter — see `.env.example`.
- **Financial mode**: budget-mode installs require `/v3/project/with-budget` while
  spend-plan-mode installs use `/v3/project`. Import tries both and pins whichever
  works, so it adapts to either mode.
- **Missing root OU**: on a freshly-wiped install with no root, the source root is
  created as a new top-level OU and everything anchors to it.

### Plan actions

`create`, `recreate`, `adopt`, `in-sync`, plus **`skipped`** (a reference couldn't
be mapped — expected, e.g. a scope with no target accounts) and **`failed`** (a real
API error on create). `skipped`/`failed` are reported separately so expected gaps
don't look like errors.

### Out of scope

Cloud accounts, cloud rules, labels, custom variables, compliance, and app-role
permission mappings are not copied (org/financial structure only).

Always read the dry-run warnings — they list every default applied and every
reference dropped before you commit with `--apply`.
